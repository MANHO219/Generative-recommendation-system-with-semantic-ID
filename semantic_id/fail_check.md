# 原方案 vs GNPR-SID V2 失败分析

## 1. 问题背景

在 LLM 微调阶段，使用原方案生成的 Semantic ID 出现了**完全失效**的情况：

- **Hit Rate = 0**：30 个生成的推荐结果中，没有任何一个命中真实访问过的 POI
- **根本原因**：码本严重坍缩，碰撞率高达 **88.3%**

---

## 2. 碰撞率分析

### 原始数据

| 指标 | 数值 |
|------|------|
| 总 POI 数 | 117,788 |
| 唯一 SID 数 | ~13,600 |
| 碰撞的 POI 数 | 103,998 (88.3%) |
| 最大冲突数 | 20 个 POI 共享同一 SID |

### 典型碰撞案例

```
SID: 53-57-39 → 19 个 POI 共享
SID: 61-16-1  → 20 个 POI 共享
```

这意味着 LLM 学到的 `<a_53><b_57><c_39>` 语义表示实际对应了 19-20 个完全不同的 POI，导致模型无法区分。

---

## 3. 架构对比

### 原方案（多源特征编码）

```
POI 原始特征
  ├─ 类别 one-hot（约 500 维，可学习嵌入压缩到 64 维）
  ├─ Plus Code 字符编码（10 维）
  ├─ 时间分布（6 维，时段比例）
  ├─ 属性：stars, is_open, review_count 等（4 维）
  └─ 地理：state_id, city_id（可选，2 维）

  → 拼接后约 256 维 → MLP Encoder([512, 256, 128]) → 128 维
  → ResidualQuantizer(3层, 每层64码)
  → Semantic ID: (code_1, code_2, code_3)
```

**问题**：
1. **特征维度过高**（256 维），信息冗余大
2. **Plus Code 字符编码**对地理位置的编码过于粗糙，相邻区域编码差异大
3. **MLP 编码器过深**，容易过拟合，泛化差
4. **码本容量不足**：64³ = 262,144 理论上足够，但特征表示差导致实际容量远低于此

---

### GNPR-SID V2（紧凑特征 + 简化编码）

```
POI 原始特征
  ├─ 类别语义：SentenceTransformer 或可学习嵌入（64 维）
  ├─ 空间：3D 球坐标 lat/lon → (x, y, z)（3 维）
  └─ 时间：Fourier 展开 6 频率 × sin/cos（12 维）

  → 直接拼接 = 79 维
  → Linear(79→64) + LayerNorm + GELU → 64 维
  → ResidualQuantizer(3层, 每层64码)
  → Semantic ID: (code_1-code_2-code_3)
```

**优势**：
1. **特征维度大幅降低**：79 维 vs 256 维，信息更紧凑
2. **球坐标**：相邻位置编码相似，地理信息连续性好
3. **Fourier 时间**：捕捉周期性模式（早/午/晚、周内分布）
4. **简化编码器**：单层 Linear，拟合能力适中，不易过拟合

---

## 4. 核心差异对比

| 维度 | 原方案 | GNPR-SID V2 |
|------|--------|-------------|
| **输入维度** | ~256 | 79 |
| **类别编码** | 可学习嵌入 (~500→64) | SentenceTransformer / 哈希 |
| **空间编码** | Plus Code 字符级 (10维) | 球坐标 3D (3维) |
| **时间编码** | 时段比例 (6维) | Fourier 展开 (12维) |
| **编码器** | MLP [512, 256, 128] | 单层 Linear + LayerNorm + GELU |
| **码本容量** | 64³ = 262,144 | 64³ = 262,144 |
| **预期碰撞率** | 88% | < 10% |

---

## 5. 失败根因分析

### 5.1 特征表示冗余

原方案的 256 维输入包含：
- 500+ 类别的 one-hot 编码 → 可学习嵌入压缩到 64 维
- Plus Code 字符（10 字符 × 1 维 = 10 维）
- 时段分布（6 维）
- 属性（4 维）

这些特征存在**大量冗余**：
- 类别嵌入维度 64，但实际类别只有几百个，信息密度低
- Plus Code 对相邻区域编码差异大（如 `7FG8V3V` 和 `7FG8V3W` 可能是相邻建筑）
- 时段分布是稀疏的 24 维向量压缩到 6 维，粒度过粗

### 5.2 码本学习失效

高维稀疏输入导致：
1. **编码器输出分布不均匀**：大部分 POI 被映射到相似的潜在向量
2. **码本坍缩**：少量码字被频繁使用，大部分码字处于"死码"状态
3. **量化残差累积误差**：三层 RQ 对高维输入的量化误差放大

### 5.3 LLM 微调灾难

当碰撞率 88% 时：
- `<a_53><b_57><c_39>` 这个 token 在 LLM 看来代表的是 19 个不同 POI 的**并集语义**
- LLM 无法从这个 token 中区分"用户去过的是哪个具体 POI"
- 推荐任务退化为"猜测用户可能去过的某个大类"

---

## 6. GNPR-SID V2 改进点

### 6.1 紧凑特征设计

| 特征 | 编码方式 | 维度 | 优势 |
|------|---------|------|------|
| 类别 | SentenceTransformer | 64 | 语义相似类别映射到相近向量 |
| 空间 | 球坐标 (x,y,z) | 3 | 相邻位置向量相似 |
| 时间 | Fourier 展开 | 12 | 捕捉周期性模式 |

### 6.2 简化编码器

原方案的多层 MLP 容易过拟合 GNPR-SID V2 的单层 Linear 表达能力适中，配合紧凑输入能达到更好的泛化效果。

### 6.3 预期效果

- **碰撞率**：88% → < 10%
- **码本利用率**：每层 > 60%
- **LLM 微调 Hit Rate**：0% → 预期 30-50%

### 7.3 GNPR-SID V2 问题分析

**问题 1：每次只训练 1 个 epoch 就中断**

日志显示 GNPR-SID V2 多次从 Epoch 0 开始，但从未超过 Epoch 1。原因可能是：
- 代码 bug 导致训练中断（可能是之前的 geo_map 或 category_vocab 相关 bug）
- 资源问题（内存/显存溢出）

**问题 2：MSE 和 CosSim 极差**

GNPR-SID V2 Epoch 0 的指标：
- MSE = 0.0435（原方案稳定后是 0.0011，**差 40 倍**）
- CosSim = 0.7818（原方案稳定后是 0.9941，**差距明显**）

这说明 GNPR-SID V2 的**特征表示有问题**，模型无法有效重构输入。可能原因：

1. **Fourier 时间特征实现错误**
   - 期望：12 维（6 频率 × sin/cos）
   - 问题：加权平均小时计算方式可能不正确

2. **球坐标特征未归一化**
   - lat/lon 范围不同，可能导致某个维度主导

3. **类别哈希嵌入效果差**
   - 没有使用 SentenceTransformer
   - 简单的哈希嵌入无法捕捉类别语义
